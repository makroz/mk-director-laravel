## Breaking changes

The 1.2.0 minor release contains four breaking changes. Each is
covered in its own subsection below with the migration steps.

### 1. `auth_users.id` is now a UUID primary key

In `1.1.x`, `auth_users.id` was an auto-incrementing integer (the
Laravel default). In `1.2.0`, it is a `CHAR(36)` UUID â€” generated by
Laravel's `HasUuids` trait or equivalent logic in your models.

**Why**: a UUID primary key is friendlier to distributed systems, multi-
region deployments, and data replication scenarios. It also makes
external-system integrations easier (no need to leak sequential
integers).

**What you need to do**:

1. Add the `HasUuids` trait to every model that extends `AuthUser`
   (`Admin`, `Member`, etc.) **before** running the migration:

   ```php
   use Illuminate\Database\Eloquent\Concerns\HasUuids;

   class Admin extends AuthUser
   {
       use HasUuids;
   }
   ```

2. Run the data migration script (see below). It will:
   - Add a temporary `uuid` column to `auth_users`.
   - Populate it with `Str::uuid()` for every row.
   - Drop the old integer `id` column.
   - Rename `uuid` to `id`.
   - Re-add the primary key constraint.

3. Verify that any external system referencing `auth_users.id` by
   integer now uses the UUID string.

**Impact**: high. This is the most invasive of the four changes and the
one that makes the upgrade effectively **irreversible** without a
backup.

### 2. Multi-tenancy is now opt-in per model

In `1.1.x`, multi-tenancy was always on for any model that used the
`HasTenantScope` trait. In `1.2.0`, tenancy must be enabled in config
**and** opted into per-model â€” by default, models do **not** have a
tenant scope.

**Why**: many consumers do not need multi-tenancy. Forcing the scope on
every consumer introduced complexity (terminating callbacks, mutex,
context lifecycle) that was not justified for single-tenant apps.

**What you need to do**:

1. Enable tenancy in `config/mk_director.php`:

   ```php
   'tenant' => [
       'enabled' => true,
       'column'  => 'client_id', // or your tenant FK column
   ],
   ```

2. For every model that should be tenant-scoped, add the trait:

   ```php
   use Mk\Director\Auth\Concerns\HasTenantScope;

   class Project extends Model
   {
       use HasTenantScope;
   }
   ```

3. Make sure your `TenantResolver` validates that the authenticated
   user actually belongs to the requested tenant. The 1.2.0 release
   adds a runtime check for this (it was a silent assumption in 1.1.x).

**Impact**: low for single-tenant apps (just leave `enabled` false).
Medium for multi-tenant apps (one config flip + per-model trait).

### 3. `MkAbility` replaces the legacy ability resolution path

In `1.1.x`, abilities were resolved inline via the `HasAbilities`
trait's `canMk()` method, which queried `ability_user` and
`ability_role` directly on every call. In `1.2.0`, ability resolution
goes through the new `Mk\Director\Auth\Services\AbilityResolver`
singleton (the `MkAbility` short-circuit) which:

- Caches results per-user for the request lifecycle.
- Short-circuits via the Sanctum token's `can()` method when the
  token has explicit `abilities` (defense-in-depth against stale DB
  state).
- Rejects empty ability lists (defense-in-depth against
  misconfiguration).

**Why**: the inline path was an N+1 query hazard in any request that
called `canMk()` more than once. Caching is the obvious fix, but it
required a singleton + Sanctum integration that did not fit cleanly in
the trait.

**What you need to do**:

1. If you have custom code that calls `$user->canMk($ability)` more
   than once per request, you can now rely on the cache â€” no code
   change needed, just expect faster responses.
2. If you have custom code that overrides `canMk()` in a model
   subclass, it will now be bypassed in favor of the resolver. To
   preserve your override, register a custom resolver via:

   ```php
   $this->app->bind(AbilityResolver::class, MyCustomResolver::class);
   ```

**Impact**: medium. The behavior change is silent (faster, cached),
but if you have custom ability logic, verify it still applies.

### 4. `ListManager` rejects unknown filter operators

In `1.1.x`, if you passed an unknown operator to `ListManager` (e.g.
`'foo'`), the code silently fell back to `=` â€” the operator you asked
for was ignored and your filter ran as if you had passed `=`. In
`1.2.0`, the same call throws `InvalidArgumentException: ListManager:
unknown filter operator "foo". Allowed: eq, neq, gt, gte, lt, lte,
like, in, not_in.`

**Why**: silent fallback hid consumer-side bugs. A typo in a filter
operator would never surface as a test failure â€” it would just return
the wrong results. The whitelist-and-throw approach makes the
misconfiguration loud.

**What you need to do**:

1. Audit any place your code (or your consumers' code) passes a filter
   operator that is not in the whitelist above. The valid operators
   are:

   | API key | SQL operator |
   |---------|--------------|
   | `eq`    | `=`          |
   | `neq`   | `!=`         |
   | `gt`    | `>`          |
   | `gte`   | `>=`         |
   | `lt`    | `<`          |
   | `lte`   | `<=`         |
   | `like`  | `LIKE`       |
   | `in`    | `IN (...)`   |
   | `not_in`| `NOT IN (...)` |

2. If you have custom operators, add them to the whitelist by
   overriding `ListManager::applyFilterOperator()` in your subclass.

**Impact**: low (most consumers don't pass custom operators). The fix
is one-line: change the operator key to the closest valid one.

## Rollback

If something goes wrong after the migration, you have three options,
in order of preference:

1. **Restore from backup** (the safest path). Stop your app, drop the
   database, restore from the snapshot you took before the upgrade,
   and roll back the composer dependency:

   ```bash
   composer require makroz/director-laravel:^1.1.0
   ```

2. **Manual SQL rollback** (only feasible if the data migration step
   hasn't been run yet). Revert the config changes, remove the
   `HasUuids` trait, and downgrade. The `bin/migrate-1.1-to-1.2.php`
   script has a `--rollback` flag for early-stage reversals.

3. **Forward fix** (only if you can identify and fix the issue without
   rolling back the migration). For example, if a consumer is passing
   an invalid filter operator, change the operator and re-run. No
   rollback needed.

In all cases: **do not skip the backup step**. Even if you plan to
"forward fix", having the backup lets you roll back without pressure.

## Migration script

The companion script `bin/migrate-1.1-to-1.2.php` automates the data
migration step (UUID primary key) and reports on the other three
breaking changes. It supports:

- `--dry-run` â€” show what would happen without changing anything.
- `--help` â€” full usage.
- `--connection=<name>` â€” run against a non-default Laravel
  connection.

The script refuses to run outside the CLI (`php_sapi_name() !== 'cli'`)
and refuses to run if `auth_users` does not exist (no-op exit code 0).

Example:

```bash
# Dry-run first
php bin/migrate-1.1-to-1.2.php --dry-run

# Real migration
php bin/migrate-1.1-to-1.2.php --connection=mysql
```

For the UUID migration, the script emits a backup reminder and an
irreversibility warning before doing anything destructive.
